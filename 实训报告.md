## 一、实训任务与目标

### 1.1 任务描述

本次综合实训要求实现一个能够解析 ELF（Executable and Linkable Format）格式文件的工具 `myreadelf`，功能上模拟 GNU `readelf`，支持：

- 显示 ELF 文件头（`-h`）、程序头/段（`-l`）、节区头部（`-S`）
- 显示符号表（`-s`）、重定位信息（`-r`）、动态段（`-d`）
- 显示 Note 段（`-n`）、版本信息（`-V`）、节组（`-g`）、架构特定信息（`-A`）
- 十六进制转储（`-x`）、符号哈希柱状图（`-I`）
- **正确解释 GOT/PLT 表内容**（`--got-plt`）
- **安全属性分析**（`--checksec`）
- 支持 ELF32/ELF64、大端/小端字节序

### 1.2 目标文件类型

| 类型 | 说明 | 代表文件 |
|------|------|---------|
| ET_EXEC / ET_DYN | 可执行文件（含PIE） | `/bin/ls` |
| ET_DYN `.so` | 共享动态库 | `/usr/lib/x86_64-linux-gnu/libc.so.6` |
| ET_REL `.o` | 可重定位目标文件 | `gcc -c foo.c -o foo.o` |

├── 节区访问

│   └── getRelocations(sec_idx) → vector<RelaInfo>

### 3.2 数据流

```
磁盘文件
  │  fread() 读入内存
  ↓
data_[0..fileSize_-1]
  │  ehdr32/64() 指针
  ↓
ELF Header → is64_, isLE_, e_shoff, e_phoff
  │
  ├─→ shdr32/64(i) → 各节区头：偏移/大小/类型
  │        │ data_[sh_offset]
  │        ↓
  │   节区原始数据：字符串表 / 符号表 / 重定位表 / 动态段 ...
  │
  └─→ phdr32/64(i) → 各程序头：类型/权限/虚拟地址
```

### 3.4 GOT/PLT 工作原理与解析

#### 延迟绑定机制（Lazy Binding）

```
程序第一次调用 printf()
        │
        ↓
   .plt 中的 printf@plt 桩代码
        │  jmp *[.got.plt + printf条目]  ← 初始值指向下一条 push
        ↓
   push <重定位索引>
   jmp PLT[0]                            ← PLT[0] = 动态链接器入口
        │
        ↓
   _dl_runtime_resolve()
        │  解析 printf 的真实地址
        ↓
   填写 .got.plt[printf条目] = printf真实地址
        │
        ↓
   第二次调用 → 直接 jmp 真实地址（无开销）
```

#### Full RELRO 的影响

启用 Full RELRO（`-z relro -z now`）后：
- 动态链接器在程序启动时**立即解析所有符号**（`DT_BIND_NOW`）
- `.got` 和 `.got.plt` 随后被标记为只读（`mprotect PROT_READ`）
- 延迟绑定被禁用，防止 GOT 覆写攻击（GOT hijacking）

`--got-plt` 选项的实现：扫描 `.rela.plt` 节区中的 `R_X86_64_JUMP_SLOT` 重定位项，按目标地址排列，同时扫描 `.rela.dyn` 中的 `R_X86_64_GLOB_DAT` 项，显示全局数据符号的 GOT 槽。

## 四、关键技术与算法

### 4.1 字节序透明处理

ELF 文件可能来自不同字节序的平台（如 ARM 大端）。本项目统一使用三个内联函数做字节序转换：

```cpp
uint32_t ELFParser::read32(const void* p) const {
    uint32_t v;
    memcpy(&v, p, 4);                  // 避免对齐问题
    if (!isLE_) v = __builtin_bswap32(v);  // 大端则交换字节序
    return v;
}
```

关键设计决策：
- 使用 `memcpy` 而非直接指针转型，避免未对齐内存访问引发 UB
- `__builtin_bswap32/64` 由 GCC/Clang 编译为单条 `bswap` 指令，零开销
- 所有字段读取都必须经过 `read16/32/64`，不允许直接解引用

### 4.2 无循环依赖的节区访问（核心架构修复）

#### 问题根源

最初的实现存在致命的循环递归：

```
getString(sec_idx, offset)
    └─→ getSection(sec_idx)          // 获取节区信息
            └─→ getSectionName(idx)  // 获取节区名
                    └─→ getString(shstrndx, name_offset)  // 需要字符串 ← 死循环！
```

调用 `./myreadelf -S /bin/ls` 时立即栈溢出，产生段错误（Segmentation Fault）。

#### 解决方案

将 `getString()`、`getSectionData()`、`getSectionName()` 三个函数改为**直接操作原始 `Elf32/64_Shdr` 指针**，完全不调用 `getSection()`：

```cpp
const char* ELFParser::getString(int sec_idx, uint32_t offset) const {
    if (!valid_ || sec_idx < 0 || sec_idx >= sectionCount()) return nullptr;

    uint32_t stype;
    uint64_t soff, ssz;

    if (is64_) {
        const Elf64_Shdr* sh = shdr64(sec_idx);   // 直接指针，无递归
        stype = read32(&sh->sh_type);
        soff  = read64(&sh->sh_offset);
        ssz   = read64(&sh->sh_size);
    } else {
        const Elf32_Shdr* sh = shdr32(sec_idx);
        stype = read32(&sh->sh_type);
        soff  = read32(&sh->sh_offset);
        ssz   = read32(&sh->sh_size);
    }

    if (stype != SHT_STRTAB) return nullptr;
    if (offset >= ssz || soff + offset >= fileSize_) return nullptr;
    return reinterpret_cast<const char*>(data_ + soff + offset);
}
```

**效果**：彻底消除循环依赖，`-S /bin/ls` 正常输出 31 个节区的完整信息。

### 4.3 悬空指针修复

`display_sections.cpp` 中的一个细微 UB：

```cpp
// 错误：临时 std::string 在 printf 调用前已析构
printf("%-17s", s.name.substr(0, 17).c_str());

// 正确：先绑定到具名变量
std::string dname = s.name.size() > 17 ? s.name.substr(0, 17) : s.name;
printf("%-17s", dname.c_str());
```

### 4.4 COMDAT 节组（SHT_GROUP）解析

C++ 模板实例化和内联函数会在多个编译单元中产生相同符号，通过 COMDAT 机制保证链接时只保留一份。解析流程：

```
扫描所有节区，找到 sh_type == SHT_GROUP 的节区
  │
  ├─→ 读取第 0 个 uint32_t：GRP_COMDAT 标志
  ├─→ sh_link 指向符号表，sh_info 是符号索引 → 获得组名
  └─→ 后续 uint32_t 数组：组内节区索引列表
```

### 4.5 GNU 属性解析（.note.gnu.property）

`.note.gnu.property` 包含 CET（Control-flow Enforcement Technology）信息：

```
Note 结构：namesz / descsz / type / name("GNU\0") / desc
  desc 中：pr_type == GNU_PROPERTY_X86_FEATURE_1_AND
           bit 0: IBT（间接分支跟踪）
           bit 1: SHSTK（影子调用栈）
```

## 五、扩展功能实现

### 5.1 安全属性扫描（`--checksec`）

对标 `checksec.sh` 工具，实现以下 6 项检测：

| 属性 | 检测方法 | 意义 |
|------|---------|------|
| **PIE** | `e_type == ET_DYN` | 地址空间随机化（ASLR）有效，防 ROP |
| **NX** | GNU_STACK 程序头无 `PF_X` | 栈不可执行，阻止 shellcode 注入 |
| **Full RELRO** | GNU_RELRO 段 + `DT_BIND_NOW` 或 `DF_1_NOW` | GOT 完全只读，防 GOT 覆写 |
| **Partial RELRO** | 仅有 GNU_RELRO 段 | 部分 GOT 保护 |
| **Stack Canary** | dynsym 中有 `__stack_chk_fail` | 编译器插入的栈保护 |
| **FORTIFY** | dynsym 中有 `__*_chk` 系符号 | 不安全 libc 函数已加固 |
| **RPATH/RUNPATH** | 动态段有 `DT_RPATH`/`DT_RUNPATH` | 库路径注入风险 |

**`/bin/ls` 检测结果**：

```
  属性           状态                 说明
  PIE              PIE enabled         ✓  地址随机化（ASLR）有效
  NX               Enabled             ✓  栈不可执行
  RELRO            Full RELRO          ✓  GOT完全只读
  Stack Canary     Found               ✓  检测栈溢出
  FORTIFY          Enabled             ✓  不安全函数已加固
  RPATH            Not set             ✓  无注入风险
  安全评分: 6/6  总体评级: 优秀 (Excellent)
```

### 5.2 文件布局可视化（`--layout`）

构建所有实体（ELF Header、PHDRs、各节区、SHDRs）的偏移/大小列表，按偏移排序后输出：
- ASCII 比例条 `[###.....]`：宽度 16 格，按文件比例填充
- 自动检测并标注**间隙（gap）**区域
- 底部统计：代码段 / 数据段 / 元数据 三类汇总大小

### 5.3 彩色输出（`-C`/`--color`）

全局标志 `bool g_color`，由 `color.h` 的宏检测后决定是否输出 ANSI 转义序列：

```cpp
#define C_GREEN  (g_color ? "\033[32m"  : "")
#define C_RESET  (g_color ? "\033[0m"   : "")
```

- 节区类型按 `SHT_*` 分组着色（PROGBITS 绿、DYNAMIC 黄、STRTAB 蓝…）
- 程序头 FLAGS 字段：R=绿、W=红、X=青
- 符号绑定：GLOBAL=白加粗、WEAK=黄、LOCAL=灰暗

## 七、问题排查与解决

### 7.1 段错误：`./myreadelf -S /bin/ls` 崩溃

**现象**：程序启动后立即崩溃，`gdb` 显示栈帧超过 1000 层。

**根本原因**：循环递归

```
getSection()
  └─ getSectionName()
       └─ getString()
            └─ getSection()  ← 死循环
```

**修复**：`getString()`、`getSectionData()`、`getSectionName()` 改为直接操作 `shdr32(idx)/shdr64(idx)` 原始指针，不再调用 `getSection()`。这是本项目最重要的架构修复。

### 7.2 `make test` FAIL：`@echo: not found`

**现象**：Make 的 `define` 宏块中使用 `@echo`（Make 特有的抑制回显语法），在 Shell 中执行时被当做命令名，导致 `@echo: not found`。

**修复**：将宏展开改为内联 `if ... then echo "[PASS]"; else echo "[FAIL]"; exit 1; fi`，完全在 Shell 语法范围内。

### 7.3 `make compare` 失败：进程替换语法

**现象**：`diff <(cmd1) <(cmd2)` 在 `/bin/sh`（dash）下报 `Syntax error: "(" unexpected`。

**修复**：在 Makefile 相应段加 `SHELL := /bin/bash`，启用 Bash 进程替换特性。

### 7.4 `ELF_ST_VISIBILITY` 未声明

部分发行版 `<elf.h>` 中该宏为 `ELF32_ST_VISIBILITY`（无通用版本）。**修复**：统一改用 `ELF32_ST_VISIBILITY(s.other)` 宏。

### 7.5 `-g` 选项注册遗漏

**现象**：`./myreadelf -g foo.o` 无输出，但 `--got-plt` 正常。

**原因**：`ELFOptions::section_groups` 字段已定义，但 `main.cpp` 的长选项表和 `switch` 中未注册 `-g`，`process_file()` 中也未调用 `display_section_groups()`。

**修复**：补全三处：长选项表中加 `{"section-groups", no_argument, nullptr, 'g'}`，switch 中加 `case 'g': opts.section_groups = true; break`，`process_file()` 中加对应调用。

### 8.2 技术收获

1. **深入理解 ELF 格式**：从字节级别理解了 ELF 头、节区头、程序头的物理布局，以及链接视图和执行视图的区别；掌握了 `.rela.plt` → GOT 槽的完整映射关系。

2. **字节序处理**：通过 `memcpy` + `__builtin_bswap` 的组合，实现了正确且高效的大小端透明访问，避免了对齐陷阱（Alignment Trap）。

3. **调试能力**：通过 `gdb` 回溯栈帧定位循环递归根因，用最小改动（改变三个函数的实现路径）彻底解决了架构性问题。

4. **安全知识**：通过实现 `--checksec`，深刻理解了 PIE、NX、RELRO、Stack Canary、FORTIFY 各自的保护机制及其检测原理；通过 `--got-plt` 理解了延迟绑定的完整流程及 Full RELRO 的防护意义。

5. **工程实践**：将功能拆分为独立的 `.cpp` 文件，降低了模块耦合；利用 `Makefile` 实现自动化测试和与权威工具的对比验证，提升了开发效率和代码可信度。

### 8.3 不足与改进方向

- 未实现 `.debug_*` 节区（DWARF 调试信息）的解析
- `--unwind`（`-u`）选项目前为空实现，EH_FRAME 解析较复杂
- 对 ARM / RISC-V 等非 x86 架构的架构特定信息（`-A`）未做针对性解析
- 可进一步实现交互式 TUI 界面（ncurses）提升可用性
